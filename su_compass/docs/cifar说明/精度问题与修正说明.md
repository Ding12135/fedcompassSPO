# CIFAR-10 精度问题诊断与修正说明

本文档记录 **2026-07 云上 CIFAR-10 主实验**（FedCompass vs q_only，budget=1500，3 seeds）中精度上不去的原因、与新方法（Oort-Compass / q_only）相关的特殊问题，以及修正路线与验收标准。

**范围说明：**

- 当前全部实验均在 **虚拟时间框架**（`su_compass`）下运行；本文 **不讨论**「虚拟时间 vs 论文 gRPC wall-clock」带来的口径差异。
- 重点回答：**在现有虚拟实验框架内**，为何 global accuracy 卡在 ~45%、为何 q_only 系统收益未转化为精度优势、以及如何修正后做出 **相对 FedCompass 有收益** 的对比。

## 2. 精度上不去的根因（按优先级）

以下原因 **FedCompass 与 q_only 共用**（baseline 问题）；§3 单独说明 **q_only 额外恶化** 的机制。

### 2.1 【P0】组聚合时 BN buffer 被错误平均

**位置：** `src/appfl/aggregator/fedcompass_aggregator.py`

ResNet-18 含大量 `BatchNorm2d`。聚合逻辑中：

- **可训练参数**（`named_parameters`）：按 FedCompass 梯度/staleness 规则更新 ✅
- **BN running_mean / running_var / num_batches_tracked**：在 **组聚合**（多客户端同时贡献）时被 **算术平均** ❌

```python
# 组聚合分支（简化）
else:  # 非 parameter key
    global_state[name] += local_model[name]
    ...
    global_state[name] = torch.div(global_state[name], len(local_models))
```

**后果：**

- 客户端上传的是 pseudo-gradient（`send_gradient=True`），BN buffer 来自各客户端 **本地 shard 上的统计量**
- 跨客户端平均后，**global eval 使用的 BN 统计量与真实全局数据分布不匹配**
- 本地 train 仍可很高（BN 在 train mode 用 batch 统计），但 **eval mode 依赖错误的 running stats** → global acc 被系统性压低

这是当前与论文 ~55% 差距 **最可疑、收益最大** 的单点实现问题。budget 从 600 提到 1500 仅带来约 **+1.6 pp**（43.88→45.48），符合「训练在涨、评估口径错了」的判断。

### 2.2 【P0】训练 batch size 与论文不一致

| 配置 | train_batch_size |
|------|------------------|
| 论文对齐配置 `examples/config/server_fedcompass_paper_cifar10_8.yaml` | **64** |
| 当前主实验 `su_compass/config/virtual_fedcompass_cifar10_8.yaml` | **128** |

batch 翻倍会改变每步梯度噪声、有效 epoch 数与收敛路径。在 FL + Non-IID 下可能进一步放大 client drift。应在 BN 修复后 **单独验证** 改回 64 的边际收益。

### 2.3 【P1】无学习率衰减，末段精度震荡

当前 SGD **lr=0.1 恒定**（与论文 Appendix 表一致），但半异步 FL 后期 staleness 与组聚合节奏复杂，容易出现：

- max acc 出现在中段
- 末段 global eval **回落**（seed2027：max=final=45.48%，但 last10=38.01%）

可选修正：在 global update 达到某阈值或虚拟时间过半后 **step decay**（如 ×0.1），目标是将 **last10 与 max 差距压到 <3 pp**。

### 2.4 【P1】Dirichlet Non-IID 下的结构性上限

数据划分：`dirichlet_noniid, alpha2=0.5`（Dual Dirichlet，与论文一致）。

- 各客户端只见到 **偏斜小分片** → 本地 train acc 极易到 99%
- global eval 在 **client_0 持有的完整 CIFAR test set（10000 张）** 上执行（`su_compass/virtual/eval.py`）
- 半异步 + staleness 使 global 曲线 **剧烈波动**（同 run 内 25%–43% 跳动属常见现象）

这不是 bug，但意味着：**不应拿 centralized 90%+ 或本地 train acc 衡量 FL 是否成功**；应对齐论文 Dirichlet 平台 **50–60%**。

### 2.5 【P2】梯度裁剪（稳定化改动）

虚拟 CIFAR 配置开启了 `clip_grad=True, clip_value=10`（论文原始 CIFAR 配置为 `clip_grad=False`）。目的是防止异常 batch 导致 NaN/权重爆炸。对收敛可能有轻微影响，优先级低于 BN 与 batch size。

### 2.6 【已排除 / 非主因】

| 项 | 说明 |
|----|------|
| staleness 多项式方向 | 已修复为 `(u+1)^(-a)` |
| 梯度裁剪顺序 | 已在 `optimizer.step()` 前 |
| budget 不足 | seed2027 已 1500 updates，max 仍 ~45%，继续加 budget 收益有限 |
| 虚拟时间口径 | **按本文档范围忽略** |

## 3. 新方法（q_only）为何尤其「精度吃亏」

Oort-Compass **q_only** 的设计目标：在 **不系统性降低最终精度** 的前提下，改善 late / deadline / 早期 TTA。

当前 CIFAR 结果：**系统侧达标倾向，精度侧未达标**。

### 3.1 机制：SystemPenalty 压低 Q → 训练量不足

q_only 将 FedCompass 的 `speed = round_time / Q` 分母替换为 **reason-aware 的 effective_step_time**（通信、迟到、波动、可用性四维惩罚叠加）。

| 效果 | seed2027 数据 |
|------|---------------|
| mean Q 下降 | 客户端 local_steps 整体更保守 |
| late / deadline 改善 | late 126 vs 196，deadline 64 vs 142 ✅ |
| mean staleness 上升 | 2.14 vs 1.41 ❌ |
| final / last10 下降 | final 38.44% vs 45.48%，last10 34.09% vs 38.01% ❌ |

**解释链：**

```text
惩罚过重 → Q 偏小 → 每轮有效训练步少
         → 聚合节奏未同步优化（group 逻辑与 baseline 相同）
         → staleness 反升
         → 末段 global 质量更差
```

MNIST 上该 trade-off 可接受（final 差 ~0.7 pp）；CIFAR + 当前 ~45% 平台下，**再减 Q 会直接掉精度**。

### 3.2 缺少 Q 下限保护（`q_floor_ratio` 未实现）

`主实验代码优化与安全实现.md` 已设计：

```text
Q_final = max(Q_oort, floor(Q_baseline × ρ))    # ρ ∈ [0.75, 0.85]
```

代码中 **`q_floor_ratio` 尚未落地**。q_only 可能把 Q 压到 baseline 以下过多，在 CIFAR 上表现为 **max 尚可、final/last10 明显落后**。

### 3.3 BN 问题对 q_only 的「放大效应」

BN 错误聚合 **同时伤害** baseline 与 q_only。但 q_only 本就减少每客户端训练量，**全局模型更依赖每次聚合的质量**；评估口径错误 + 训练量偏少 → **final 差距被放大**（seed2027 差 7 pp，大于 max 差 0.5 pp）。

### 3.4 q_and_group（暂不主跑）

本地 seed2026：max **39.59%**，mean staleness **5.54**，精度与稳定性均差于 q_only。**当前阶段不作为新方法对比对象**。

## 5. 修正路线与预期效果

**原则：一次只改一个因素；单 seed 门禁通过后再扩 3 seeds；baseline 先上去，再调 q_only。**

### 5.1 阶段 A：抬升 FedCompass 平台（共用基础）

| 步骤 | 改动 | 文件 | 预期 |
|------|------|------|------|
| **A1** | 组聚合 **不平均 BN buffer**，保留全局原有 buffer（或仅用聚合后权重、eval 前不混 buffer） | `src/appfl/aggregator/fedcompass_aggregator.py` | max **+5~10 pp**，有机会 **≥50%** |
| **A2** | `train_batch_size` / `val_batch_size` **128→64** | `su_compass/config/virtual_fedcompass_cifar10_8.yaml` | 更贴论文，可能 **+1~3 pp** |
| **A3** | （可选）LR step decay | trainer 或 runner | last10 与 max 差距 **<3 pp** |

**A1 快速验证（单 seed，budget=600 即可初判）：**

```bash
cd /root/gpufree-data/fedcompass
source /opt/conda/etc/profile.d/conda.sh && conda activate app
source su_compass/output/cifar10_gpu_calibration.env

python -m su_compass.experiments.run_virtual_fl \
  --algorithm fedcompass \
  --server_config su_compass/config/virtual_fedcompass_cifar10_8.yaml \
  --client_config examples/config/client_cifar10.yaml \
  --num_clients 8 --num_global_epochs 600 --seed 2026 \
  --base_step_time "$BASE_STEP" --model_size_mb "$MODEL_MB" --update_size_mb "$MODEL_MB" \
  --output_dir su_compass/output/cifar10_fix_smoke/bn_fix/seed2026
```

**A1 门禁：** max acc **≥ 50%** → 进入 A2；仍 <50% → 先查 BN 改法是否完整（单客户端路径、general_buffer 合并路径均需覆盖）。

**A 阶段完成标准（fedcompass，budget=1500，单 seed）：**

- max acc **≥ 50%**（理想 **≥55%**）
- TTA@50 **可达**
- last10 与 max 差距 **<3 pp**

### 5.2 阶段 B：让 q_only 相对 FedCompass 有收益

在 **阶段 A 完成后**，FedCompass 平台应接近论文 Dirichlet 区间；再调 q_only，目标恢复文档叙事：

> 系统指标更好，且 **last10 不低于 baseline 超过 1 pp**。

| 步骤 | 改动 | 说明 |
|------|------|------|
| **B1** | 实现 `q_floor_ratio`（建议默认 **0.80**，0=禁用） | `su_compass/virtual/algorithms/oort_compass.py` + 配置 |
| **B2** | 降低惩罚强度 | 扫描 `λ_late=0.5`, `λ_comm=0.5`, `penalty_clip=5.0` |
| **B3** | 单 seed 对比 | 同 seed、同 budget、同 A 阶段配置 |

**B 阶段验收（相对 FedCompass，3 seed 均值）：**

| 指标 | 目标 |
|------|------|
| last10 accuracy | q_only ≥ fedcompass **−1 pp** |
| max accuracy | q_only ≥ fedcompass **−1 pp**（或持平） |
| late uploads | 相对 baseline **↓ ≥25%** |
| deadline triggers | 相对 baseline **↓ ≥25%** |
| TTA@50 或 TTA@40 | q_only **≤** fedcompass（至少 2/3 seeds） |

### 5.3 阶段 C：正式重跑主实验矩阵

修正通过门禁后，**删除或 `--force` 重跑** 旧 budget=600 的 seed2026 目录，统一：

```text
fedcompass + q_only × seeds 2026/2027/2028 × budget=1500
```

分析：

```bash
python -m su_compass.experiments.analyze_cifar10_main \
  --output_root su_compass/output/cifar10_main
```

## 6. 如何判断「相对 FedCompass 有收益」

### 6.1 论文叙事下的「有收益」

CIFAR Dirichlet 主对比 **不是**「谁 final 最高」，而是：

1. **在相近 top accuracy 平台下**（last10 差距 ≤1 pp）
2. **更快达到目标精度**（TTA@50 为主，TTA@40 为辅）
3. **更少 late / deadline**（系统负担更低）

因此：**先让 FedCompass 到 50%+，再比 q_only**；在 45% 平台上比 TTA 意义有限（两者都到不了 TTA@50）。

### 6.2 当前数据下的结论（修正前）

| 维度 | q_only vs fedcompass（seed2027） | 是否有收益 |
|------|----------------------------------|------------|
| max acc | 44.96% vs 45.48%（−0.5 pp） | 基本持平 |
| final / last10 | 明显更低 | **无精度收益** |
| late / deadline | 明显更低 | **有系统收益** |
| TTA@40 | 2024 vs 1127（更慢） | **无效率收益**（平台太低时 TTA 不可比） |
| TTA@30 | 335 vs 472（更快） | **有早期收益**（仅说明早期冲得快，末段掉队） |

**修正前不宜主张「q_only 全面优于 FedCompass」**；可主张「reason-aware 惩罚能改善 late/deadline，但需 Q 下限与 baseline 平台修复才能保住精度」。

### 6.3 修正后期望图表形态

```text
accuracy-time 曲线：
  - 两者均在虚拟时间 ~1000–2000s 进入 50%+ 区间（TTA@50 可定义）
  - q_only 到达 50% 的 virtual_time 更短（左移）
  - 末段 last10 两条曲线间距 ≤1 pp

系统指标：
  - q_only 的 late、deadline 柱状图显著低于 fedcompass
  - mean staleness 不应比 baseline 恶化超过 ~0.2
```
