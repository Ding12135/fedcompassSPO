# Oort-Compass 后续优化方向

本文档面向论文发表目标，说明当前 `Oort-Compass` 方向的研究定位、已有进展、后续优化路线与可行性判断。配套实验设计见同目录下 `实验方案.md`。

---

## 1. 论文目标定位

当前工作不应表述为“提出全新的联邦学习优化算法”，而应定位为：

```text
Utility-Guided FedCompass Scheduling for Heterogeneous Semi-Asynchronous Federated Learning
```

中文可表述为：

```text
面向异构半异步联邦学习的效用导向 FedCompass 调度扩展。
```

核心思想是将 Oort 的 `statistical utility / system cost` 从原始的“客户端选择”场景，适配到 FedCompass 的三个决策位置：

| 决策位置 | FedCompass baseline | Oort-Compass 优化目标 |
|----------|---------------------|-----------------------|
| Q 分配 | 根据 `speed = round_time / Q` 估计单步耗时 | 用多维系统代价修正 effective step time |
| arrival group 分组 | 根据 expected / latest arrival time 加入 group | 避免高风险客户端加入时间余量不足的 group |
| 聚合权重 | 使用 staleness-aware 聚合 | 后续可叠加 utility-aware 权重 |

论文中的贡献点建议收敛为三条：

1. **Utility adaptation**：将 Oort 的效用导向思想从 client selection 适配为 FedCompass 中的 Q 分配与 arrival group 调度。
2. **Reason-aware system cost**：将 FedCompass 的单一 `speed` 扩展为通信、迟到、波动、可用性组成的多维系统代价。
3. **Trace-driven validation**：基于真实本地训练 + 虚拟时间事件驱动框架，系统评估精度-时间、迟到、deadline、staleness 与 Q 分布。

## 2. 当前进展与阶段性结论

### 2.1 已完成能力

当前代码已经具备以下基础能力：

- 使用真实 GPU 训练生成本地模型更新。
- 使用 `VirtualRuntimeModel` 生成虚拟下载、训练、上传、波动与可用性等待时间。
- 使用 `RuntimeStateTracker` 维护端侧多维状态。
- 复刻 FedAvg、FedAsync、FedCompass 的虚拟时间控制器。
- 新增 `VirtualOortCompassController`，支持 `shadow`、`q_only`、`q_and_group` 模式。
- 输出 `scheduler_trace`、`aggregation_trace`、`group_trace`、`round_reports`、`state_trace`、`oort_trace` 与 `global_eval_trace`。

### 2.2 已完成实验结论

已完成 `MNIST + 8 clients + budget=250 + seeds 2026/2027/2028` 的 FedCompass vs `q_only` 对比。

主要结果：

| 指标 | FedCompass | q_only | 结论 |
|------|------------|--------|------|
| final accuracy | 97.21% | 96.49% | q_only 无稳定最终精度优势 |
| max accuracy | 98.48% | 98.41% | 两者基本持平，MNIST 存在天花板效应 |
| TTA@95 | 81.1 | 62.4 | q_only 更快到达 95% |
| late uploads | 34.7 | 19.3 | q_only 约减少 44% |
| deadline triggers | 24.3 | 16.3 | q_only 约减少 33% |
| mean staleness | 1.19 | 1.37 | q_only staleness 略升 |
| mean Q | 129.5 | 115.5 | q_only 更保守地分配本地步数 |

阶段性判断：

- **已验证**：reason-aware Q 分配能稳定改善系统行为，尤其是 late 与 deadline。
- **部分验证**：q_only 能更快达到较高精度阈值，尤其是 TTA@95。
- **尚未验证**：最终精度稳定超过 FedCompass。
- **当前短板**：staleness 略升，且 `statistical_utility()` 仍为常数 1，尚未体现 Oort 的统计效用优势。

因此，目前结果适合支撑“系统调度增强有效”，但还不足以支撑“全面优于 FedCompass”。

## 3. 后续优化路线

后续优化应围绕论文发表目标分为四条主线：补齐调度闭环、增强统计效用、放大任务难度、补充消融与鲁棒性。

### 3.1 方向一：从 q_only 推进到 q_and_group

**目标**：解决 q_only 降低迟到但可能增加 staleness 的问题。

当前 q_only 只改变 Q：

```text
Q = remaining_time / effective_step_time
```

它能减少高风险客户端的本地步数，但 group 选择仍继承 FedCompass baseline。后续应启用 `q_and_group`，让风险较高的客户端避免加入时间余量不足的 group。

预期收益：

- late uploads 继续下降或保持低位。
- deadline triggers 继续下降。
- mean staleness 持平或下降。
- TTA@95 不退化。

论文价值：

- 证明 Oort-Compass 不只是“少给慢客户端 Q”，而是完整影响半异步调度中的 group formation。
- 补齐 Stage B / Gate 3 的方法链条。

可行性：

- 代码已经实现 `q_and_group` 开关。
- 只需按同 seed、同 budget 跑实验。
- 是当前投入产出比最高的下一步。

### 3.2 方向二：接入 loss-based statistical utility

**目标**：从 system-only Oort-Compass 升级为完整 utility-guided Oort-Compass。

当前 `statistical_utility()` 固定返回 1：

```text
OortScore_i = 1 / SystemCost_i
```

后续应将本地训练 loss 或验证 loss 接入状态追踪器，形成：

```text
OortScore_i = StatisticalUtility_i / SystemCost_i
```

推荐形式：

```text
StatisticalUtility_i = clip(normalize(local_loss_i), u_min, u_max)
SystemCost_i = 1 + λ_comm * comm_ratio
               + λ_late * late_rate
               + λ_var * step_time_cv
               + λ_avail * (1 - availability_rate)
```

设计原则：

- loss 必须归一化，避免单个高 loss 客户端长期主导调度。
- utility 应使用滑动平均，避免单轮异常造成调度震荡。
- 初期只让 statistical utility 影响 Q / group，不直接改聚合权重。

预期收益：

- 在更强 Non-IID 或更难数据集上，高 loss 客户端能获得更多训练机会。
- 精度侧有机会稳定优于 FedCompass，而不仅是系统指标改善。
- 支撑论文中 “utility-guided” 的完整论证。

可行性：

- 需要改动 `client_runtime.py`、trace 字段、`state.py` 或 Oort tracker。
- 预计 1-2 天可实现第一版，后续需要 2-3 轮消融调参。
- 是论文方法完整性的关键工作。

### 3.3 方向三：切换到更难数据集与更强 Non-IID

**目标**：避免 MNIST 天花板效应，让方法差异在收敛曲线中更明显。

MNIST 上两种方法最高精度都达到 98%+，精度空间太小，容易出现 final accuracy 随最后几次异步聚合波动的现象。后续需要引入更难任务：

| 数据集 / 设置 | 目的 | 预期观察 |
|---------------|------|----------|
| CIFAR-10 | 放大模型与数据难度 | TTA@70/80/85 更有区分度 |
| 更强 Dirichlet Non-IID | 放大客户端统计差异 | loss utility 更可能带来精度收益 |
| 更多客户端 | 放大调度压力 | late、deadline、staleness 差异更明显 |
| stress runtime profile | 放大系统异构 | reason-aware penalty 的收益更明显 |

论文价值：

- 回应“MNIST 太简单”的质疑。
- 证明方法不只在玩具数据上有效。
- 支撑 generalization 与 robustness claim。

可行性：

- 项目已有 CIFAR-10 相关配置，代码复用性高。
- 主要成本是训练时间增加与 budget 重新校准。
- 建议先做 CIFAR-10 8 clients，再考虑 16 clients 或更强 stress profile。

### 3.4 方向四：utility-aware aggregation 作为扩展消融

**目标**：探索 OortScore 是否应参与聚合权重，而不仅参与调度。

可选形式：

```text
weight_i ∝ staleness_fn(staleness_i) * normalized(OortScore_i)
```

该方向不建议过早作为主方法，因为它会同时改变调度和聚合，导致归因困难。建议在 Q/group 证据充分后作为扩展实验：

- baseline: FedCompass aggregation
- + Q only
- + Q and group
- + Q/group and utility-aware aggregation

论文价值：

- 如果有效，可作为 full Oort-Compass 的增强版本。
- 如果无效，也可作为 negative result，说明主要收益来自调度而非聚合。

可行性：

- 需要改动聚合器调用或 wrapper。
- 风险较高，建议排在 loss utility 之后。

## 4. 可发表性判断

### 4.1 当前状态

仅凭当前 MNIST q_only 结果，适合形成技术报告或 workshop preliminary result，但不宜直接投稿完整论文。原因是：

- 精度没有稳定优于 FedCompass。
- q_only 的 staleness 略升。
- 尚未接入 Oort 的 statistical utility。
- 数据集难度不足，容易被质疑 MNIST 饱和。

### 4.2 完成后续实验后的可行性

如果后续完成 `q_and_group + CIFAR-10 + loss utility + 消融`，并达到以下结果：

- TTA@目标精度稳定优于 FedCompass。
- final / max / last-k accuracy 不系统性低于 FedCompass。
- late 与 deadline 持续显著下降。
- staleness 在 `q_and_group` 下持平或下降。
- loss utility 在强 Non-IID 或 CIFAR-10 上带来额外收益。

则可支撑一篇系统 / 应用 / workshop 方向论文。

建议目标：

| 投稿目标 | 可行性 | 适合叙事 |
|----------|--------|----------|
| arXiv / 技术报告 | 很高 | 完整记录框架、方法和实验 |
| 联邦学习 / 边缘计算 workshop | 中高 | Oort-guided semi-async scheduling |
| 应用型会议 / 中文核心 | 中高 | 异构端侧调度优化与实验验证 |
| 顶会主会 | 较低 | 需要更强理论、更多 baseline 与真实部署 |

### 4.3 最低可投稿标准

建议在正式写论文前满足以下最低标准：

| 维度 | 最低标准 |
|------|----------|
| 主结果 | 至少一个难数据集上 TTA@目标精度 3/3 seeds 优于 FedCompass |
| 精度 | final 或 last-k accuracy 均值不低于 FedCompass 超过 1pp |
| 系统指标 | late 或 deadline 至少下降 25% |
| staleness | q_and_group 下持平或下降 |
| 消融 | 至少包含 q_only、q_and_group、system+loss、w/o penalty |
| 复现 | 每个主实验至少 3 seeds，保存完整 trace |

## 5. 推荐执行顺序

后续工作建议按照论文证据链执行，而不是只追求单次精度最高。

```text
Step 1: 固化当前 MNIST fedcompass vs q_only 结果
Step 2: 跑 MNIST q_and_group，验证 staleness 与 late 是否进一步改善
Step 3: 跑 CIFAR-10 fedcompass / q_only / q_and_group，观察难任务上的 TTA
Step 4: 接入 loss-based statistical utility
Step 5: 在 MNIST strong Non-IID 与 CIFAR-10 上比较 system-only vs system+loss
Step 6: 做 λ 消融与 stress runtime profile
Step 7: 视结果决定是否加入 utility-aware aggregation
```

推荐优先级：

| 优先级 | 工作 | 原因 |
|--------|------|------|
| P0 | q_and_group 主实验 | 代码已具备，直接补齐 Gate 3 |
| P0 | 分析脚本增加 max / last-k / TTA 指标 | 避免 final accuracy 误判 |
| P1 | CIFAR-10 对比 | 解决 MNIST 天花板质疑 |
| P1 | loss utility | 补齐 Oort 方法核心 |
| P2 | λ 消融 | 支撑 ablation |
| P2 | stress profile | 支撑 robustness |
| P3 | utility-aware aggregation | 可作为扩展，不宜过早主打 |

## 6. 写作风险与规避

| 风险 | 可能质疑 | 规避方式 |
|------|----------|----------|
| 创新不足 | 只是把 Oort 搬到 FedCompass | 强调 client selection 到 Q/group scheduling 的适配差异 |
| MNIST 太简单 | 精度提升无意义 | 加 CIFAR-10 与更强 Non-IID |
| 虚拟时间不真实 | 不是多机部署 | 明确真实训练 + 虚拟系统异构，定位为 controlled evaluation |
| 指标选择偏置 | 只报有利指标 | 同时报告 accuracy、TTA、late、deadline、staleness |
| final accuracy 波动 | 异步尾部不稳定 | 报 max、last-k mean 与完整 accuracy-time curve |
| 归因不清 | 同时改太多 | 按 q_only、q_and_group、system+loss、aggregation 分阶段消融 |

---

## 7. 阶段性结论

当前方向具备论文潜力，但主张必须克制：

- 可以主张：**reason-aware Oort-Compass 在异构半异步 FL 中能显著改善调度效率，并更快达到目标精度阈值。**
- 暂不应主张：**Oort-Compass 在最终精度上全面优于 FedCompass。**

后续最关键的验证点是：

1. `q_and_group` 是否能补齐 staleness 短板。
2. CIFAR-10 / 强 Non-IID 是否能放大 TTA 与精度差异。
3. loss-based statistical utility 是否能带来稳定精度收益。

只有这三点成立，论文才能从“系统调度增强”升级为“完整 utility-guided semi-asynchronous FL 方法”。
