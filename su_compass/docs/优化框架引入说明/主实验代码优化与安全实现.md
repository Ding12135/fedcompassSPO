- 实验矩阵：`实验方案.md`

## 1. 为什么要先做代码优化再放大实验

### 1.1 已有主实验结论（MNIST, 8 clients, budget=250, 3 seeds）

| 指标 | FedCompass | q_only | 判断 |
|------|------------|--------|------|
| final accuracy | 97.21% | 96.49% | q_only 精度略降（约 -0.73pp） |
| max accuracy | 98.48% | 98.41% | 基本持平 |
| TTA@95 | 81.1 | 62.4 | q_only 明显更快 |
| late uploads | 34.7 | 19.3 | q_only 约减 44% |
| deadline triggers | 24.3 | 16.3 | q_only 约减 33% |
| mean staleness | 1.19 | 1.37 | q_only 略升 |
| mean Q | 129.5 | 115.5 | q_only 更保守 |

**已证实**：reason-aware Q 分配能改善系统侧指标（late、deadline、TTA@95）。

**未解决**：

1. `q_only` 以精度换系统收益，final accuracy 未稳定优于 FedCompass。
2. `q_and_group` 在 budget=30 下与 `q_only` 的 dispatch 决策 **31 条完全一致**，分组优化实际未生效。
3. 若直接上云跑 CIFAR / 更多 seeds，会把上述问题放大为大量无效算力消耗。

### 1.2 优化目标（本阶段）

在进入图像主实验前，必须让：

```text
q_only_v2     ：保留 late/deadline 收益，缩小 final accuracy 差距
q_and_group_v2：相对 q_only 进一步改善 staleness，且不牺牲精度
```

并满足 **安全引入原则**：每一步改动可 shadow 验证、可单 seed 门禁、可回退。

## 2. 理论支撑：Oort 思想如何映射到 FedCompass

### 2.1 Oort 原始机制（client selection 场景）

Oort（OSDI'21）在同步 FL 中做 **客户端选择**，核心效用形式为：

```text
StatisticalUtility_i  ∝  |B_i| · sqrt(mean(loss_i²))
SystemCost_i          ∝  round_duration_i
OortScore_i           =  StatisticalUtility_i / SystemCost_i
```

对超过时长容忍阈值的 straggler，还会施加 duration 惩罚（utility 按 `(T/duration)^α` 衰减）。

**关键**：Oort 不是“把所有客户端都压慢”，而是 **识别高代价客户端并降低其被选中概率**；统计效用高的客户端仍应获得更多训练机会。

### 2.2 FedCompass 场景的差异

FedCompass 是 **半异步** 调度，不做 client selection，所有客户端都参与。Oort 思想需适配到两个决策点：

| Oort 原始决策 | FedCompass 适配位置 | 当前实现 |
|---------------|---------------------|----------|
| 选不选客户端 | 不丢弃客户端 | 全部参与 |
| duration 惩罚 | Q 分配分母 | `effective_step_time = speed × SystemPenalty` |
| 避免 straggler 拖慢轮次 | arrival group 过滤 | `q_and_group` 模式（当前有 bug） |
| statistical utility | 暂未接入 | `statistical_utility() ≡ 1` |

因此，本阶段优化应聚焦 **SystemCost 的正确性与温和性**，以及 **group 决策的客户端时间语义**，而不是急于接入 loss utility。

### 2.3 FedCompass baseline 的结构性问题（优化动机）

FedCompass 用单一标量：

```text
speed_i = round_time_i / Q_i
```

`round_time` 混合了计算、通信、波动、不可用等待。结果是：

- **network_poor** 客户端：通信占比高（实测 comm_ratio ≈ 0.8），却被整体 speed 惩罚，Q 分配缺乏“慢因解释性”。
- **compute_volatile** 客户端：波动大，仍可能被分进 deadline 较紧的 group → late 与 staleness 同时恶化。

Oort-Compass 的理论价值在于：**把 speed 拆成 reason-aware 的 SystemPenalty**，并在 group 层阻止“高风险 + 时间不够”的组合。

### 2.4 当前实验现象的理论解释

| 现象 | 理论解释 |
|------|----------|
| q_only 降低 late/deadline | SystemPenalty 增大 effective_step_time → Q 减小 → 上传更轻、更易赶上 latest |
| q_only 降低 final accuracy | 惩罚过重且四维同时叠加 → mean Q 从 129 降到 115，训练量不足 |
| q_only 升高 staleness | Q 减小但 group 选择未变，客户端仍可能进入时间紧张的 group，聚合节奏未优化 |
| q_and_group 与 q_only 完全相同 | 实现 bug：group 过滤条件对 FedCompass 所有 group 恒为 false（见 §3.1） |

## 3. 已定位的实现问题

### 3.1 q_and_group 分组过滤恒不生效（P0 bug）

当前 `_group_too_risky()`（`su_compass/virtual/algorithms/oort_compass.py`）使用：

```text
slack = latest_arrival - expected_arrival
span  = expected_arrival - created_time
过滤条件：(slack / span) < slack_min_ratio
```

对 FedCompass 默认 `latest_time_factor = 1.2` 的新建 group：

```text
slack / span = (λ - 1) / 1 = 0.2    （与客户端、当前时刻无关的常数）
slack_min_ratio 默认 = 0.1
→ 0.2 < 0.1 恒为 false → 永远不过滤
```

**结论**：不是“q_and_group 理论无效”，而是 **过滤判据使用了 group 固有比例而非客户端剩余时间**，导致 mode 形同虚设。

**验证证据**（budget=30, seed2026）：

- `q_only` 与 `q_and_group` 的 `dispatch_decision_trace.csv` 31 条记录逐条相同。
- `oort_trace.csv` 中 `group_filter_applied` 相关逻辑从未改变最终 dispatch。

### 3.2 q_only 惩罚过重（P1 调参/改公式）

当前 `system_penalty()`（`utility.py`）对 comm、late、var、avail **无阈值地线性叠加**：

```text
penalty = 1 + λ_comm·comm + λ_late·late + λ_var·cv + λ_avail·(1-avail)
```

对 `network_poor` 画像，comm_ratio 长期 ≈ 0.8，即使 late_rate 不高也会吃到接近 `penalty_clip=3.0` 的上限，Q 被系统性压低。

这与 Oort “**仅对超过容忍度的 straggler 惩罚**”不一致。

### 3.3 实验效率问题（P2，非算法但影响规模）

`examples/config/client_mnist.yaml` 中 `do_validation: True`，每轮本地训练后跑完整验证集，但 **调度器不读取该结果**。虚拟实验 wall-clock 时间中这部分为纯开销。

## 5. 优化 A：修复 q_and_group 分组过滤

### 5.1 理论支撑

FedCompass 的 late 来自两类原因：

1. **Q 过大**：单轮训练时间超过 group 的 latest arrival（q_only 通过减 Q 缓解）。
2. **group 选择不当**：客户端加入时，距 `latest_arrival` 的剩余时间已不足以安全完成 `min_local_steps`（q_and_group 应解决）。

Oort 在 selection 场景会排除“预计超时”的客户端；映射到 FedCompass 即：**高风险客户端不应 join 时间余量不足的 group**，而应 create 新 group 或等待更宽松窗口。

### 5.2 安全改法（客户端时间语义）

将 `_group_too_risky(group_idx, state)` 改为 **客户端-组联合判据**：

```text
输入：
  curr_time, group.latest, group.expected
  q_divisor = effective_step_time（q_only/q_and_group 下）
  min_local_steps, risk_score(state)

判据：
  if risk <= risk_threshold: 放行

  time_to_latest = latest - curr_time
  min_needed     = min_local_steps × q_divisor × safety_factor

  if time_to_latest < min_needed:  视为 too_risky，跳过该 group
```

推荐默认：`safety_factor = 1.15`（在 min Q 基础上留 15% 时间余量）。

**为何安全**：

- 只影响 `q_and_group` mode；`shadow` / `q_only` 行为不变。
- 低 risk 客户端路径与 FedCompass 完全一致。
- 过滤失败时 fall through 到 `_create_group()`，与 baseline 相同，不会死锁。

### 5.3 涉及文件

| 文件 | 改动 |
|------|------|
| `su_compass/virtual/algorithms/oort_compass.py` | 重写 `_group_too_risky()`；`_join_group` 传入 `q_divisor` |
| `su_compass/virtual/algorithms/utility.py` | `OortConfig` 增加 `risk_safety_factor` |
| `su_compass/virtual/trace.py` | `oort_trace` 增加 `group_filter_applied` 字段 |

### 5.4 预期效果

| 指标 | 相对 q_only | 机制 |
|------|-------------|------|
| late | 持平或略降 | 少进必迟到 group |
| staleness | **应下降** | 减少“勉强入组→迟到→陈旧聚合” |
| final accuracy | 持平或略升 | 少做无效高压训练 |
| dispatch 与 q_only 差异 | **必须 > 0** | 证明 mode 生效 |

## 6. 优化 B：q_only 定向惩罚（超阈值才罚）

### 6.1 理论支撑

Oort 惩罚的是 **超过容忍 duration 的 straggler**，而非静态标签。FedCompass 的 RuntimeStateTracker 已区分：

- `late_rate`：历史迟到比例
- `communication_ratio_mean`：通信占比
- `step_time_cv`：计算波动
- `availability_rate`：可用性

定向惩罚 = **谁在某维度“确实有问题”才罚谁**，避免 stable_fast 客户端被 comm 维度误伤。

### 6.2 安全改法

在 `system_penalty()` 中改为带阈值的超额惩罚：

```text
penalty = 1.0

if late_rate > τ_late:
    penalty += λ_late · (late_rate - τ_late)

if comm_ratio > τ_comm:
    penalty += λ_comm · (comm_ratio - τ_comm)

if step_cv > τ_var:
    penalty += λ_var · (step_cv - τ_var)

if availability < τ_avail:
    penalty += λ_avail · (τ_avail - availability)

penalty = clip(penalty, 1.0, penalty_clip)
```

**推荐初始阈值**（与 8 客户端画像校准一致）：

| 参数 | 建议值 | 依据 |
|------|--------|------|
| τ_late | 0.15 | 低于此视为正常波动 |
| τ_comm | 0.50 | network_poor 才显著超 0.5 |
| τ_var | 0.30 | compute_volatile 才显著超 0.3 |
| τ_avail | 0.90 | availability_limited 画像 |

**为何安全**：

- `min_reports` 冷启动仍保留：前几轮 penalty=1。
- 阈值可通过 CLI / `OortConfig` 暴露，便于消融。
- 旧行为可通过 `τ=0` 复现，支持 A/B 对比。

### 6.3 预期效果

| 指标 | 相对当前 q_only | 相对 FedCompass |
|------|-----------------|-----------------|
| late | 略升但仍显著低于 baseline | ↓ 25%+ |
| final accuracy | **应提升** | 差距缩至 < 0.5pp |
| mean Q | 略升（少误罚） | 仍低于 baseline |
| TTA@95 | 可能略回退 | 仍应优于 baseline |

## 7. 优化 C：Q 下限保护（温和修正）

### 7.1 理论支撑

FedCompass 的 `speed` 已是 EMA 平滑后的经验估计，代表“历史平均单步耗时”。Oort 适配应是：

```text
Q_oort = floor(remaining / effective_step_time)     # 惩罚后
Q_final = max(Q_oort, floor(Q_baseline × ρ))        # 不低于 baseline 的一定比例
```

其中 `Q_baseline = floor(remaining / speed)`，`ρ ∈ [0.75, 0.85]`。

这对应“**在 speed 估计基础上做有界修正**”，而非完全替换 FedCompass 调度逻辑。

### 7.2 安全改法

在 `_join_group` / `_create_group` 计算 `assigned_steps` 后：

```text
q_base = floor(remaining / speed)
q_oort = floor(remaining / eff_step)
q_capped = max(q_oort, floor(q_base * q_floor_ratio))
assigned_steps = clip(q_capped, min_local_steps, max_local_steps)
```

`OortConfig` 新增 `q_floor_ratio: float = 0.80`，`0` 表示禁用（回退纯 q_oort）。

**为何安全**：

- 只限制“最多压低多少”，不提高超过 baseline 的 Q。
- 与优化 B 正交，可独立开关做消融。

### 7.3 预期效果

- final accuracy：**几乎一定改善**（减少过度保守 Q）。
- late：仍低于 FedCompass（惩罚仍起作用，只是有底）。

## 8. 优化 D：惩罚渐进启用（加强冷启动）

### 8.1 理论支撑

Oort 在线学习客户端 duration/utility，早期样本不足时不应激进调度。当前 `min_reports=2` 后立即全量惩罚，窗口初期 `late_rate`、`comm_ratio` 方差大，易造成前期 Q 震荡并传导为 cascade late。

### 8.2 安全改法

```text
blend = min(1.0, (num_reports - min_reports) / warmup_reports)
effective_penalty = 1.0 + (penalty - 1.0) * blend
```

推荐：`min_reports=2`，`warmup_reports=5`。

**为何安全**：

- 前 `min_reports` 轮与 FedCompass 完全一致。
- `warmup_reports` 内逐步过渡，shadow 模式可验证前若干轮 trace 对齐。

## 10. 安全平稳实现流程（Gate 机制）

### 10.1 四段式引入（与代码 mode 对齐）

```text
Stage 0  shadow     只算分数，不改 dispatch（已有，用于回归）
Stage 1  q_only_v2  优化 B (+ 可选 C/D)，单 seed 门禁
Stage 2  q_and_group_v2  优化 A，在 q_only_v2 基础上单 seed 门禁
Stage 3  3 seeds    MNIST budget=250 正式对比
Stage 4  CIFAR smoke → 定 budget → 图像主实验
```

**禁止跳步**：未过 Stage 2 门禁，不得跑 Stage 3 的 q_and_group 三 seeds。

### 10.2 单 seed 门禁（seed2026）

#### Gate-1：q_and_group 行为生效（budget=30 即可）

```text
✓ dispatch_decision_trace 与 q_only 至少 1 处不同
✓ oort_trace 中 group_filter_applied > 0
✓ shadow 模式仍与 fedcompass baseline 一致（Stage A 回归）
```

#### Gate-2：q_only_v2 指标平衡（budget=250）

```text
✓ late ≤ fedcompass × 0.75
✓ final_acc ≥ fedcompass - 0.5pp
✓ TTA@95 ≤ fedcompass × 0.85
```

#### Gate-3：q_and_group_v2 相对 q_only_v2（budget=250）

```text
✓ mean_staleness ≤ q_only_v2
✓ late ≤ q_only_v2 × 1.05（允许略升 5%）
✓ final_acc ≥ q_only_v2 - 0.3pp
```

### 10.3 推荐验证命令

```bash
# Gate-1: q_and_group 修复验证（快）
python -m su_compass.experiments.run_virtual_fl \
  --algorithm oort_compass --oort_mode q_and_group \
  --seed 2026 --num_global_epochs 30 \
  --client_config examples/config/client_mnist_virtual.yaml \
  --output_dir su_compass/output/gate/q_and_group_v2/seed2026

# Gate-2: q_only_v2
python -m su_compass.experiments.run_virtual_fl \
  --algorithm oort_compass --oort_mode q_only \
  --seed 2026 --num_global_epochs 250 \
  --client_config examples/config/client_mnist_virtual.yaml \
  --output_dir su_compass/output/gate/q_only_v2/seed2026

# 分析
python -m su_compass.experiments.analyze_convergence_compare
```

### 10.4 回退策略

| 情况 | 处理 |
|------|------|
| Gate-1 失败 | 只改 `_group_too_risky`，不进入 Gate-2/3 |
| Gate-2 late 不达标 | 略增 λ_late 或降低 τ_late，不增加 seeds |
| Gate-2 accuracy 不达标 | 启用/增大 `q_floor_ratio`，或降低 penalty_clip |
| Gate-3 staleness 未改善 | 调 `risk_threshold` / `safety_factor`，仍不跑 3 seeds |

## 14. 总结

当前主实验说明：**Oort 式 SystemPenalty 进入 Q 分配是有效的**，但实现上存在两处关键缺口：

1. **q_and_group 分组过滤判据错误**，导致 mode 完全无效（P0 bug）。
2. **q_only 惩罚过重且缺乏阈值**，以精度换取系统指标（P1 公式问题）。
