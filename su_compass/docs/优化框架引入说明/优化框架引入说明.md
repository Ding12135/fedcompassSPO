# Oort 优化框架引入说明

本文档说明后续如何在当前 `su_compass` 虚拟时间联邦学习框架中引入 **Oort** 的效用调度思想，并将其适配到 FedCompass 的 Q 分配、arrival group 分组和后续聚合策略中。

文档目标不是直接复现 Oort 官方代码，而是给出一套可落地的 **Oort-guided FedCompass** 改进路线，后续代码实现、实验对比和论文方法描述均以本文为依据。

---

## 1. 当前基线与问题

当前 `su_compass` 已完成以下基础能力：

- 使用真实 GPU 训练生成 `local_model`。
- 使用 `VirtualRuntimeModel` 根据端侧画像生成虚拟 `finish_time`。
- 使用 `RuntimeStateTracker` 记录端侧运行状态。
- 复刻 FedAvg / FedAsync / FedCompass 虚拟时间控制器。
- 输出 `scheduler_trace`、`aggregation_trace`、`dispatch_decision_trace`、`group_trace`、`global_eval_trace` 等 trace。

其中 FedCompass baseline 的核心调度依据仍然是：

```text
speed_i = round_time_i / local_steps_i
```

也就是说，当前 FedCompass 主要根据客户端的历史响应速度分配 Q 和 arrival group。

### 1.1 FedCompass 当前调度逻辑

FedCompass 控制器主要位于：

```text
su_compass/virtual/algorithms/fedcompass.py
```

关键策略如下：

| 方法 | 当前行为 | 主要依据 |
|------|----------|----------|
| `_record_info()` | 更新客户端速度估计 | `round_time / Q` |
| `_join_group()` | 尝试加入已有 arrival group | `remaining_time / speed` |
| `_create_group()` | 创建新 arrival group | `speed`、`Qmin/Qmax` |
| `_group_aggregation()` | 组聚合 | `staleness`、`local_steps` |
| `_record_dispatch_decision()` | 记录调度决策 | 当前只记录部分端侧状态 |

### 1.2 当前不足

FedCompass 当前的问题不是不能处理异构，而是它把所有慢因压缩进一个 `speed`：

```text
round_time = train_time + download_time + upload_time + spike_delay + availability_wait
```

这样会导致：

| 场景 | FedCompass 可能的问题 |
|------|----------------------|
| 计算慢 | 用 speed 调 Q 基本合理 |
| 通信慢 | 可能被误判为整体慢，Q 分配不一定合理 |
| 可用性差 | 迟到风险没有提前进入分组决策 |
| 波动大 | 可能进入 deadline 较紧的 group |
| 高重要度客户端 | 当前没有统计重要度建模 |

因此，当前优化方向是：

> 在 FedCompass 的 speed-aware 调度基础上，引入 Oort 的 utility-guided 思想，把「端侧系统状态」和「端侧统计重要度」共同纳入 Q 分配、分组和聚合决策。

## 2. Oort 的定位

Oort 是 OSDI 2021 的联邦学习客户端选择框架：

```text
Oort: Efficient Federated Learning via Guided Participant Selection
```

Oort 不是 FedAvg / FedCompass 这种完整训练算法，也不是 PSO / GA 这类群优化算法。它更准确的定位是：

```text
utility-guided participant selection framework
```

即：用客户端效用指导联邦学习中的参与方选择。

### 2.1 Oort 原始思想

Oort 将客户端效用拆成两部分：

```text
统计效用 statistical utility
系统代价 system cost
```

简化理解：

```text
OortScore_i = StatisticalUtility_i / SystemCost_i
```

其中：

- `StatisticalUtility_i`：客户端数据对全局模型的潜在贡献，Oort 原文常用训练 loss 近似。
- `SystemCost_i`：客户端参与训练带来的系统时间代价，Oort 原文主要考虑训练完成时间。

原版 Oort 的主要用途是：

```text
从大量 cross-device 客户端中选择哪些客户端参与本轮训练。
```

### 2.2 与当前场景的差异

当前项目是 cross-silo 风格的 8 客户端 FedCompass 实验：

| 维度 | Oort 原始场景 | 当前项目场景 |
|------|---------------|--------------|
| 客户端规模 | 大规模 cross-device | 8 客户端 cross-silo |
| 决策对象 | 选择客户端 | 分配 Q、加入 group、聚合 |
| 系统代价 | 粗粒度 round duration | 可拆分的端侧状态 |
| 调度框架 | client selection | FedCompass arrival group |
| 端侧状态 | 较少 | `RuntimeStateTracker` 已记录多维状态 |

因此不能直接把 Oort 官方代码作为插件接入，而应适配其 utility 思想：

```text
Oort client selection
    ↓
Oort-guided FedCompass Q/group scheduling
```

## 3. 拟引入的方法：Oort-Compass

后续建议将新方法命名为：

```text
Oort-Compass
```

论文中可表述为：

```text
Utility-Guided FedCompass Scheduling
```

核心定义：

> Oort-Compass 将 Oort 的客户端效用建模从「选择客户端」适配为 FedCompass 中的「Q 分配、arrival group 分组和可选聚合权重调整」。

### 3.1 总体设计

Oort-Compass 引入两个核心分数：

```text
SystemCost_i        # 系统代价
StatisticalUtility_i # 统计重要度
OortScore_i          # 综合效用
```

计算关系：

```text
OortScore_i = StatisticalUtility_i / SystemCost_i
```

在第一版中，可以先只引入系统侧：

```text
StatisticalUtility_i = 1
OortScore_i = 1 / SystemCost_i
```

这相当于 reason-aware FedCompass。

后续加入端侧重要度后：

```text
StatisticalUtility_i = f(local_loss, data_size, update_drift)
OortScore_i = StatisticalUtility_i / SystemCost_i
```

## 4. SystemCost：如何使用端侧状态

当前 `RuntimeStateTracker` 已经维护了多维端侧状态：

```text
step_time_mean
step_time_cv
compute_step_time_mean
communication_time_mean
communication_ratio_mean
availability_wait_mean
availability_wait_ratio_mean
late_rate
early_margin_mean
q_min_hit_rate
q_max_hit_rate
availability_rate
last_staleness
```

这些状态位于：

```text
su_compass/runtime/state.py
```

### 4.1 推荐的系统代价公式

第一版建议采用可解释的乘性惩罚：

```text
SystemCost_i =
    round_time_mean_i
    × (1 + λ_comm × communication_ratio_mean_i)
    × (1 + λ_late × late_rate_i)
    × (1 + λ_var  × step_time_cv_i)
    / max(availability_rate_i, ε)
```

各项含义：

| 项 | 含义 |
|----|------|
| `round_time_mean_i` | 客户端最近窗口平均完成时间 |
| `communication_ratio_mean_i` | 通信慢因 |
| `late_rate_i` | 历史迟到风险 |
| `step_time_cv_i` | 速度波动风险 |
| `availability_rate_i` | 可用性风险 |

推荐默认参数：

```text
λ_comm = 0.5
λ_late = 1.0
λ_var  = 0.5
ε      = 1e-6
```

后续可做参数消融。

### 4.2 为什么不用单纯 speed

FedCompass 的 `speed` 是总耗时归一化：

```text
speed_i = round_time_i / Q_i
```

它无法区分：

- 计算慢
- 通信慢
- 可用性等待
- 偶发波动

Oort-Compass 的 `SystemCost` 则允许调度器区分慢因，使调度决策更可解释。

## 5. StatisticalUtility：后续端侧重要度

端侧重要度不是第一版必须完成，但应在设计上预留。

### 5.1 最小重要度定义

Oort 原版主要使用客户端 loss 作为统计效用代理。后续建议先采用：

```text
StatisticalUtility_i = normalized(local_loss_i)
```

若有本地样本数：

```text
StatisticalUtility_i =
    normalized(local_loss_i) × sqrt(num_samples_i / mean_num_samples)
```

### 5.2 扩展重要度定义

后续可加入模型更新漂移：

```text
StatisticalUtility_i =
    w_loss  × norm(local_loss_i)
  + w_data  × norm(num_samples_i)
  + w_drift × norm(update_drift_i)
```

其中：

```text
update_drift_i = 1 - cosine(local_update_i, reference_update)
```

或：

```text
update_drift_i = ||local_model_i - global_model||_2
```

第一版不建议直接加入过多重要度项，避免变量太多导致实验解释困难。

### 5.3 当前代码缺口

当前 trace 已经记录了系统状态，但结构化 loss 还没有完整进入调度器：

```text
round_reports.csv 目前没有 local_train_loss / local_val_loss 字段
```

后续若做完整 Oort-Compass，需要补充：

```text
local_train_loss
local_val_loss
num_train_samples
update_drift
```

## 6. OortScore 如何用于 FedCompass 调度

### 6.1 只改 Q 分配

这是第一阶段最推荐的接入方式，风险最低。

FedCompass 当前：

```text
Q_i = floor(remaining_time / speed_i)
```

Oort-Compass：

```text
effective_step_time_i = speed_i × SystemPenalty_i
Q_i = floor(remaining_time / effective_step_time_i)
```

其中：

```text
SystemPenalty_i =
    1
  + λ_comm × communication_ratio_mean_i
  + λ_late × late_rate_i
  + λ_var  × step_time_cv_i
```

直观解释：

- 通信差：降低 Q，避免把网络慢客户端拖进更重任务。
- 迟到多：降低 Q 或避免紧 deadline。
- 波动大：降低进入紧 group 的概率。
- 可用性差：调度更保守。

### 6.2 改 group 选择

FedCompass 当前 `_join_group()` 的选择标准是：

```text
在满足 [Qmin, Qmax] 的 group 中，选 local_steps 最大的 group。
```

Oort-Compass 可以改为：

```text
JoinScore(g, i) =
    local_steps_i
  - μ_risk × DeadlineRisk_i
  + μ_score × OortScore_i
```

其中：

```text
DeadlineRisk_i =
    late_rate_i
  + step_time_cv_i
  + communication_ratio_mean_i
```

第一版更稳妥的规则：

```text
如果客户端 DeadlineRisk 高，则跳过 slack 太小的 group。
```

这里的 slack 可定义为：

```text
slack_g = latest_arrival_time_g - expected_arrival_time_g
```

### 6.3 改聚合权重

FedCompass 当前聚合权重由 `FedCompassAggregator` 内部结合 `staleness_fn` 处理。

后续可以扩展成：

```text
weight_i ∝ staleness_fn(staleness_i) × OortScore_i
```

但第一版不建议立即改聚合器。原因：

- Q / group 调度已经足够形成第一阶段创新。
- 同时改 Q、group、aggregator 会让实验变量过多。
- 可以先把 `OortScore` 写入 trace，后续再做 aggregation ablation。

只负责 Oort utility 计算，不负责事件循环和聚合。

class OortUtilityTracker:

## 8. 推荐实现阶段

### 阶段 A：System-only Oort-Compass

目标：先证明端侧状态进入调度后能改善 FedCompass。

特征：

```text
StatisticalUtility_i = 1
OortScore_i = 1 / SystemCost_i
```

改动：

- 新增 `utility.py`
- 新增 `oort_compass.py`
- 修改 `_join_group()` / `_create_group()` 中的 Q 计算
- trace 增加 system cost 与 q_before/q_after

验证指标：

- `global_eval_trace`：同虚拟时间下 accuracy
- `dispatch_decision_trace`：Q 分配是否更稳定
- `group_trace`：deadline 触发比例、late_clients 是否下降
- `aggregation_trace`：平均 staleness 是否下降

### 阶段 B：加入端侧重要度

目标：完成 Oort 原始 statistical utility 的适配。

特征：

```text
StatisticalUtility_i = norm(local_loss_i)
OortScore_i = StatisticalUtility_i / SystemCost_i
```

改动：

- 结构化记录 `local_train_loss` 或 `local_val_loss`
- 将 loss 更新到 `OortUtilityTracker`
- 用 `OortScore` 参与 Q/group 决策

验证指标：

- 高 loss 客户端是否获得更合理的训练机会
- 同预算下 global accuracy 是否提升
- 重要客户端的 staleness 是否降低

### 阶段 C：Utility-aware aggregation

目标：把 OortScore 从调度扩展到聚合权重。

特征：

```text
weight_i ∝ staleness_fn(staleness_i) × OortScore_i
```

改动：

- 新增或包装 aggregator
- trace 记录 per-client aggregation weight

验证指标：

- global accuracy
- loss 曲线稳定性
- 高 staleness 低 utility 更新是否被抑制

## 10. 论文描述建议

### 10.1 方法定位

推荐表述：

```text
We adapt Oort's utility-guided participant selection principle to FedCompass,
where the utility is used for local workload assignment and arrival-group
scheduling instead of client selection.
```

中文：

```text
本文将 Oort 的效用导向客户端选择思想适配到 FedCompass 半异步调度中，
将其从「选择客户端」扩展为「分配本地训练步数 Q 与 arrival group」。
```

### 10.2 创新点

可写成三点：

1. **Utility adaptation**：将 Oort 的 `statistical utility / system cost` 从客户端选择适配为 FedCompass 的 Q/group 调度。
2. **Reason-aware system cost**：利用 `RuntimeStateTracker` 将系统代价从粗粒度 round time 扩展为通信、迟到、波动、可用性组成的多维代价。
3. **Trace-driven validation**：基于 `su_compass` 的虚拟时间 trace 验证精度、staleness、Q 分布和 group 行为。

### 10.3 避免的表述

不建议写：

```text
本文提出一种全新的联邦学习训练算法 Oort。
```

应写：

```text
本文提出一种 Oort-guided FedCompass 调度扩展。
```

## 11. 风险与注意事项

### 11.1 不要一次改太多

第一版只建议改 Q/group，不改 aggregator。

否则结果提升或下降时难以判断原因来自：

- Q 分配
- group 分配
- 聚合权重
- loss 重要度

### 11.2 统计重要度需要谨慎

`loss_i` 作为重要度时，可能导致高 loss 客户端被过度偏向。建议使用归一化和裁剪：

```text
StatisticalUtility_i = clip(norm(loss_i), u_min, u_max)
```

### 11.3 系统惩罚需要保持可解释

不要一开始引入复杂神经网络或强化学习。当前项目的优势是 trace 清晰、机制可解释，适合先做规则型 Oort-Compass。

## 12. 建议的最终落地顺序

```text
1. 新增 utility.py：实现 SystemCost / OortScore
2. 新增 oort_compass.py：复制 FedCompass baseline 并替换 Q 计算
3. run_virtual_fl.py 增加 --algorithm oort_compass

5. 跑 MNIST 30 budget 对比 FedCompass

7. 补 local_loss，升级 full Oort-Compass
8. 做消融实验
9. 视结果决定是否做 utility-aware aggregation
```

## 13. 当前结论

当前项目已经验证了虚拟端侧建模和 FedCompass 虚拟调度的可用性。下一步引入 Oort 时，应将 Oort 作为一个 **utility-guided scheduling principle**，而不是直接移植其官方 client selection 代码。

最合理的工程路线是：

```text
FedCompass speed-aware scheduling
    ↓
System-only Oort-Compass
    ↓
System + Statistical Utility Oort-Compass
    ↓
Utility-aware aggregation（可选）
```

这样既能保持 FedCompass baseline 的可比性，也能逐步证明端侧状态与端侧重要度对 Q 分配和分组调度的优化效果。

## 14. 稳步安全引入方案（如何保证不破坏 baseline 且可验证有效）

本节回答两个工程问题：**如何稳步安全地引入 Oort**，以及**如何确认引入确实有效**。

核心原则：先做成「可开关、可对照、baseline 不变」的旁路，再逐步让它真正影响调度。任何一步变差都能回退到上一步。

### 14.1 引入分为 4 个 gate，每步可回滚

```text
Gate 0  baseline 冻结      → 固定 FedCompass 结果作为对照基准
Gate 1  影子模式(只算不用) → 计算 Oort 分数，写 trace，不改 Q
Gate 2  单点生效(只改 Q)   → 用开关控制，可随时退回 baseline
Gate 3  分组生效           → Q 稳定后再改 group
```

每个 gate 通过验证后才进入下一个；任何 gate 变差，退回上一 gate。

### 14.2 Gate 0：冻结 baseline

先用固定 seed 跑一次 FedCompass，作为后续所有对比的基准，不写任何 Oort 代码：

```text
su_compass/output/baseline_fedcompass_seed2026/
  ├── global_eval_trace.csv       # accuracy-time 基准
  ├── aggregation_trace.csv       # staleness 基准
  ├── group_trace.csv             # late / deadline 基准
  └── dispatch_decision_trace.csv
```

### 14.3 Gate 1：影子模式（shadow）——最关键的安全设计

新增 `utility.py` 与 `oort_compass.py`，但**第一版让 Oort 分数只算不用**：

- `oort_compass.py` 继承 `VirtualFedCompassController`。
- `_join_group` / `_create_group` 仍用原 `speed` 算 Q（与 baseline 完全一致）。
- 额外计算 `system_cost` / `oort_score` / `q_if_oort`，**只写入 trace**。

验证标准（必须成立）：

```text
oort_compass(shadow) 的 global_eval / aggregation / group trace
        == baseline_fedcompass 完全一致
```

因为 Q 仍是旧公式，结果必须逐行相同。这一步证明「接入没有破坏 FedCompass」。

同时可在 `dispatch_decision_trace` 中离线确认 Oort 想调整的方向是否合理：

```text
q_baseline    # 当前真正使用的 Q
q_if_oort     # Oort 会给的 Q
system_cost
oort_score
```

例如 `network_poor` 应满足 `q_if_oort < q_baseline`，方向合理后再进 Gate 2。

### 14.4 Gate 2：让 Oort 只影响 Q（带开关）

引入配置开关：

```text
oort_mode: off | shadow | q_only | q_and_group
oort_lambda_comm / oort_lambda_late / oort_lambda_var
```

`q_only` 时才真正用 `q_if_oort` 覆盖 Q，**group 逻辑仍不动**。

有效性判定（与 Gate 0 基准比，同 seed、同 budget）：

| 指标 | 期望方向 | 来源 |
|------|----------|------|
| 平均 / 最大 staleness | 下降或持平 | `aggregation_trace` |
| late_clients 比例 | 下降 | `group_trace` |
| deadline 触发比例 | 下降 | `group_trace` |
| 相同虚拟时间下 accuracy | 不变差 | `global_eval_trace` |
| network_poor 的 q_max_hit_rate | 下降 | `state_trace` |

接受标准建议：staleness / late 改善，且 accuracy 不下降超过阈值（如不掉 1%）。达不到就调 λ 或退回 shadow。

### 14.5 Gate 3：让 Oort 影响分组

Q 稳定后再改 `_join_group` 的 group 选择（见第 15 节）。用 `q_and_group` 开关控制，可随时退回 `q_only`。

### 14.6 如何确定「引入有效」——三重保证

1. **一致性回归**：shadow 模式必须与 baseline 完全一致 → 证明未破坏。
2. **单变量对比**：一次只开一个开关（q_only → q_and_group）→ 提升可归因。
3. **多 seed 复现**：2–3 个 seed 都改善才算真有效，避免单次波动。

### 14.7 gate 与落地步骤对照

| 步骤 | 做什么 | 开关 | 验证 |
|------|--------|------|------|
| 1 | 冻结 baseline | — | 存基准 trace |
| 2 | `utility.py` + shadow | `oort_mode=shadow` | trace 与 baseline **逐行一致** |
| 3 | Q 生效 | `oort_mode=q_only` | staleness / late 下降，acc 不降 |
| 4 | 分组风险过滤 | `oort_mode=q_and_group` | late / deadline 下降 |
| 5 | 多 seed 复现 | — | 2–3 seed 均改善 |
| 6 | （后续）重要度 / 聚合权重 | 新开关 | acc-time 提升 |

## 15. Q 与分组的调度决策逻辑（引入后如何调度）

本节说明引入 Oort 后，具体如何用端侧状态做 **Q 分配** 与 **分组** 决策。

### 15.1 第一步：端侧状态压成系统代价 SystemCost

数据全部来自 `RuntimeStateTracker.snapshot(client_id)`：

```text
SystemCost_i = round_time_mean_i
             × (1 + λ_comm · communication_ratio_mean_i)
             × (1 + λ_late · late_rate_i)
             × (1 + λ_var  · step_time_cv_i)
             / max(availability_rate_i, ε)
```

再折算成「有效单步时间」，便于直接替换 speed：

```text
effective_step_time_i = speed_smoothed_i × SystemPenalty_i

SystemPenalty_i = 1
                + λ_comm · communication_ratio_mean_i
                + λ_late · late_rate_i
                + λ_var  · step_time_cv_i
```

- 第一版 `StatisticalUtility_i = 1`，即纯 system 版；语义：通信差 / 迟到多 / 波动大的客户端被视为「更慢」，Q 给得更保守。
- **冷启动保护**：`num_reports < 2` 时 `SystemPenalty_i = 1`，退化为原始 FedCompass，避免前几轮无数据乱调。

### 15.2 第二步：Q 决策（改动最小，最先做）

现状（`_join_group` / `_create_group`）：

```text
Q = floor(remaining_time / speed_i)
```

改为：

```text
Q = floor(remaining_time / effective_step_time_i)
Q = clip(Q, Qmin, Qmax)
```

关键：**只把分母从 `speed` 换成 `effective_step_time`，其余 group 遍历与截断逻辑完全不动**，风险最小，并可在 `dispatch_decision_trace` 中对比 `q_baseline` 与 `q_after_oort`。

预期方向：

| 客户端 | penalty | Q 变化 | 效果 |
|--------|---------|--------|------|
| network_poor（comm_ratio 高） | 大 | 变小 | 上传负担轻、少迟到 |
| compute_volatile（step_cv 高） | 大 | 变小 | 少触发 deadline |
| stable_fast | ≈1 | 基本不变 | 与 baseline 一致 |

### 15.3 第三步：分组决策（Q 稳定后再做）

#### 方案 A：迟到风险门槛（推荐，改动小）

现状 `_join_group`：在满足 `[Qmin, Qmax]` 的 group 中选 `local_steps` 最大的。

增加风险过滤——高风险客户端不进 slack 太小的 group：

```text
slack_g = latest_arrival_time_g - expected_arrival_time_g
risk_i  = late_rate_i + step_time_cv_i + communication_ratio_mean_i

# 遍历 group 时：
if risk_i > risk_threshold and slack_g < slack_min:
    continue   # 跳过该 group
```

含义：波动大 / 爱迟到的客户端只进「时间余量宽」的组，直接压低 `late_clients` 和 deadline 触发比例。

#### 方案 B：综合评分选组（可选，变量多）

```text
JoinScore(g, i) = Q_ig - μ_risk · risk_i · (1 / slack_g)
```

选 `JoinScore` 最大的 group。建议方案 A 验证有效后再尝试 B。

### 15.4 第四步：聚合权重（暂不做）

先不碰 `FedCompassAggregator`。等 Q + 分组均验证有效后再考虑：

```text
weight_i ∝ staleness_fn(staleness_i) × oort_score_i
```

同时改会导致变量过多，难以归因。

### 15.5 每次 CLIENT_UPLOAD 的完整决策流程

```text
1. _record_info：更新 speed（不变）
2. runtime_state = tracker.snapshot(client_id)
3. 算 SystemPenalty_i / effective_step_time_i           # utility.py
4. _assign_group:
     _join_group:
        - 遍历 group
        - Q = remaining / effective_step_time_i          # 15.2
        - （可选）风险过滤 slack                          # 15.3 A
        - 选 Q 最大（或 JoinScore 最大）的 group
     失败 → _create_group（同样用 effective_step_time）
5. 写 dispatch_decision_trace:
     q_baseline / q_after_oort / system_cost / oort_score / system_penalty
```

### 15.6 小结

- **安全引入**：`baseline 冻结 → shadow(只算不用，必须与 baseline 一致) → q_only → q_and_group`，全程配置开关可回退；有效性靠「一致性回归 + 单变量对比 + 多 seed」三重保证。
- **调度决策**：端侧状态压成 `SystemCost_i`，折成 `effective_step_time_i`，**Q 用它替换 speed 作分母**；**分组用迟到风险门槛**（高风险客户端不进 slack 小的组）；聚合权重留到最后再动。
