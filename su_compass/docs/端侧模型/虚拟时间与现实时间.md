# 虚拟时间与现实时间

## 1. 当前结论

当前阶段先不改 FedCompass 主代码。

我们先独立验证端侧状态模型：

```text
ClientRuntimeProfile
-> VirtualRuntimeModel
-> ClientRoundReport
-> RuntimeStateTracker
```

等端侧状态验证通过后，再考虑把虚拟时间接入 `CompassScheduler`。

## 2. 两种时间的含义

### 2.1 现实时间

现实时间是程序真实运行消耗的时间。

例如：

```text
一张 RTX 4090 上真实训练一个客户端模型花了 0.4 秒。
```

现实时间主要用于：

- 程序运行耗时统计。
- GPU 资源使用分析。
- 调试训练性能。

现实时间不适合直接表示端侧异构，因为所有客户端都在同一张 GPU 上训练时，训练速度会过于接近。

### 2.2 虚拟时间

虚拟时间是模拟出来的联邦学习系统时间。

例如：

```text
同一个客户端虽然在 4090 上真实训练 0.4 秒，
但根据端侧 profile，它在真实端侧设备上可能需要 5.2 秒才完成训练和上传。
```

虚拟时间主要用于：

- 模拟不同客户端的计算能力。
- 模拟不同客户端的网络带宽。
- 模拟运行波动、卡顿和可用性变化。
- 驱动 arrival group、迟到率和 safe-Q 等调度逻辑。

## 3. 为什么需要虚拟时间

当前实验环境只有一张 4090。

如果所有客户端都直接用真实时间，FedCompass 看到的是：

```text
client_0 快
client_1 快
client_2 快
...
```

很难形成以下端侧场景：

- 稳定慢客户端。
- 波动快客户端。
- 网络差客户端。
- 可用性受限客户端。
- 重要但较慢客户端。

所以需要通过虚拟时间把同一张 GPU 上的真实训练结果映射成不同端侧设备的完成时间。

## 4. 推荐实验模式

后续推荐使用：

```text
真实训练 + 虚拟端侧时间
```

含义是：

```text
真实训练：
    4090 负责真实训练模型，得到 local_model 或 gradient。

虚拟时间：
    端侧模型根据 client profile 生成 virtual_finish_time。

调度处理：
    FedCompass/SU-Compass 使用 virtual_finish_time 判断到达顺序、迟到和分组。
```

这样可以保证：

- 模型更新是真实训练得到的。
- 系统异构由端侧 profile 控制。
- 单张 GPU 也能模拟多个异构端侧客户端。

## 5. 不建议的方式

不建议通过真实 `sleep` 等待虚拟时间。

例如不建议：

```text
真实训练完成
sleep(virtual_round_time)
再上传服务器
```

原因：

- 实验会变慢。
- GPU 会空等。
- 大量客户端时效率很差。
- 现实时间和虚拟时间容易混在一起。

更推荐后续使用虚拟事件队列。

## 6. 虚拟事件队列

虚拟事件队列的核心思想是：

```text
真实训练完成后，不立即按真实完成顺序处理。
而是为每个客户端计算 virtual_finish_time。
然后按 virtual_finish_time 从小到大处理更新。
```

示例：

```text
client_0 真实训练完成时间：0.4s，virtual_finish_time：2.0s
client_1 真实训练完成时间：0.4s，virtual_finish_time：5.5s
client_2 真实训练完成时间：0.4s，virtual_finish_time：3.2s
```

服务器处理顺序应为：

```text
client_0 -> client_2 -> client_1
```

而不是 GPU 上的真实训练顺序。

## 7. 后续主代码最小改动原则

当前先不改主代码。

后续如果接入虚拟时间，主代码建议只保留一个很小的入口：

```text
curr_time = kwargs.get("virtual_time", real_time)
```

更稳的方式是封装为：

```text
self._now(**kwargs)
```

这样：

```text
没有 virtual_time 时，原始 FedCompass 继续使用真实时间。
有 virtual_time 时，SU-Compass 实验使用虚拟时间。
```

## 8. 需要统一使用虚拟时间的地方

一旦接入虚拟时间，以下调度相关量都必须使用虚拟时间：

- `dispatch_time`
- `finish_time`
- `client_start_time`
- `expected_arrival_time`
- `latest_arrival_time`
- `late_rate`
- `arrival error`
- `group waiting time`
- `time-to-target`

不能一部分用现实时间，一部分用虚拟时间。

## 9. 需要特别注意的 FedCompass 逻辑

当前 `CompassScheduler` 中有真实时间和真实定时器逻辑。

后续接入虚拟时间时需要重点检查：

- `_record_info()`
- `_assign_group()`
- `_join_group()`
- `_create_group()`
- `_group_update()`
- `_group_aggregation()`
- `threading.Timer`

其中 `threading.Timer` 是最需要谨慎处理的部分。

在虚拟时间实验中，真实 timer 不适合直接表示 arrival group 到期，后续可能需要改成事件驱动或手动推进虚拟时钟。

## 10. 当前阶段不做什么

当前阶段只验证端侧状态，不做以下改动：

- 不改 `CompassScheduler`。
- 不改 `NaiveTrainer`。
- 不改 `FedCompassAggregator`。
- 不接入虚拟事件队列。
- 不改变原始 FedCompass 的调度行为。

## 11. 后续接入顺序

推荐后续顺序：

```text
Phase 1:
    独立验证 RuntimeProfile 和 RuntimeStateTracker。

Phase 2:
    写虚拟事件队列，在 su_compass 内部验证 virtual_finish_time 排序。

Phase 3:
    给 CompassScheduler 增加最小 virtual_time 入口。

Phase 4:
    替换或绕开真实 threading.Timer，使用虚拟事件驱动 arrival group。

Phase 5:
    接入 safe-Q 和 group-Q 决策。
```

可以这样说明实验时间建模：

```text
All local model updates are trained on a single RTX 4090 GPU, while client runtime heterogeneity is emulated using FedScale/FLASH-style virtual runtime profiles. The scheduler observes virtual dispatch and finish times to reproduce heterogeneous client arrival behavior.
```

中文表述：

```text
所有客户端模型更新均在单张 RTX 4090 上真实训练得到；客户端计算、网络和可用性异构通过 FedScale/FLASH 风格的虚拟运行画像建模。调度器使用虚拟派发时间和虚拟完成时间来模拟异构端侧客户端的到达行为。
```
