# 端侧模型

## 1. 当前阶段目标

本阶段先独立建立端侧运行模型，不接入 FedCompass 主代码。

目标是先验证：

- 能否通过可调参数生成稳定快、稳定慢、计算波动、网络差、可用性受限等端侧场景。
- Runtime State 是否能从虚拟运行反馈中识别这些端侧状态。
- 后续是否可以把同一套 `ClientRoundReport -> RuntimeStateTracker` 接入 `CompassScheduler`。

## 2. 开源与论文依据

本模块不是凭空构造，也不是直接复制某个开源仓库。

参考关系如下：

- `profile.py` 参考 **FedScale / FLASH** 的端侧系统异构建模思想，包括计算能力、网络带宽、运行波动和客户端可用性。
- `virtual_runtime.py` 参考 **FedScale** 的计算/通信时间建模思想，将训练时间拆成本地训练、下载、上传、网络延迟、系统波动和可用性等待。
- `state.py` 参考 **Oort** 的 feedback-driven client modeling 思想，以及当前 **FedCompass** 中 `speed = client_update_time / client_steps` 的客户端速度估计逻辑。
- 本项目没有直接照搬 Oort/FedScale/FLASH 代码，而是按 FedCompass 的 arrival group 和 Q 调度目标做轻量重写。

推荐论文表述：

```text
The runtime profile follows the system heterogeneity modeling used in FedScale and FLASH, including compute capacity, network bandwidth, runtime variability, and client availability. The server-side state update follows the feedback-driven client modeling idea of Oort and extends FedCompass's speed-aware scheduler toward richer runtime state tracking.
```

## 3. 模块结构

```text
su_compass/runtime/
├── profile.py          # 端侧画像参数：计算能力、网络带宽、波动、卡顿、可用性
├── virtual_runtime.py  # 不使用 sleep 的虚拟运行时间模型
├── state.py            # Runtime State 记录与滑动窗口更新
└── simulator.py        # 独立模拟验证入口
```

## 4. 端侧画像参数

`ClientRuntimeProfile` 是单客户端的端侧运行画像。

主要参数：

- `compute_capacity`：计算能力，越大训练越快。
- `compute_jitter_cv`：计算时间波动系数，越大越不稳定。
- `upload_bandwidth_mbps`：上传带宽。
- `download_bandwidth_mbps`：下载带宽。
- `network_jitter_cv`：网络波动系数。
- `base_latency`：基础网络延迟。
- `spike_probability`：偶发卡顿概率。
- `spike_delay_mean`：偶发卡顿平均时长。
- `availability_probability`：客户端单轮可用概率。

## 5. 虚拟运行时间

当前不使用简单 `sleep`，而是使用虚拟时间计算：

```text
round_time =
    download_time
  + train_time
  + upload_time
  + spike_delay
  + availability_wait
```

其中：

```text
train_time = local_steps × base_step_time / compute_capacity
download_time = model_size / download_bandwidth
upload_time = update_size / upload_bandwidth
```

计算、网络和可用性都可以加入随机波动。

## 6. Runtime State

每轮虚拟运行生成 `ClientRoundReport`，再由 `RuntimeStateTracker` 更新状态。

### 6.1 端侧每轮需要上报的内容

当前阶段先不改 FedCompass 主代码，但端侧模型已经按后续接入主代码的形式定义了每轮上报内容。

一条 `ClientRoundReport` 对应客户端完成一次完整参与：

```text
服务器下发/客户端拉取模型
  -> 下载模型
  -> 本地训练 local_steps 步
  -> 上传更新
  -> 服务器收到更新
```

因此，端侧每轮应向服务端反馈以下信息：

```text
ClientRoundReport
├── 基础身份与时间
│   ├── client_id
│   ├── dispatch_time
│   ├── finish_time
│   └── local_steps
├── 慢因拆分耗时
│   ├── train_time
│   ├── download_time
│   ├── upload_time
│   ├── spike_delay
│   └── availability_wait
├── 调度目标与结果
│   ├── target_arrival_time
│   ├── latest_arrival_time
│   ├── hit_q_min
│   ├── hit_q_max
│   └── staleness
└── 可用性状态
    └── available
```

字段含义：

- `client_id`：客户端编号。
- `dispatch_time`：服务端下发任务或模型的时间；主代码接入时对应客户端被 `_assign_group()` 分配下一轮 Q 时记录的 `start_time`。
- `finish_time`：服务端收到客户端上传更新的时间；主代码接入时对应 `schedule()` 被客户端上传触发时的当前时间。
- `local_steps`：本轮客户端本地训练步数 Q，不是联邦轮数。
- `train_time`：本轮纯本地训练耗时，用于判断是否计算慢。
- `download_time`：本轮下载模型耗时，用于判断下行通信开销。
- `upload_time`：本轮上传模型更新耗时，用于判断上行通信开销。
- `spike_delay`：本轮偶发卡顿耗时，用于判断突发系统慢因。
- `availability_wait`：本轮不可用等待耗时，用于判断设备可用性慢因。
- `target_arrival_time`：目标聚合组期望到达时间。
- `latest_arrival_time`：目标聚合组允许的最晚到达时间。
- `hit_q_min` / `hit_q_max`：本轮 Q 是否命中调度器设置的下界/上界。
- `staleness`：本轮更新相对当前全局模型的陈旧度。
- `available`：本轮客户端是否处于可用状态。

### 6.2 为什么要加入慢因拆分耗时

原 FedCompass 主要依赖：

```text
speed = client_update_time / client_steps
```

这个值只能说明客户端整体快慢，不能说明慢在哪里。对于后续是否等待、放入哪个 arrival group、下一轮分配多少 Q，仅知道 `round_time / local_steps` 不够，因为下面几种慢客户端需要不同策略：

- 计算慢但稳定：可以分配更少 Q 或放入更宽松的到达组。
- 网络慢：通信固定开销高，可能更适合增大 Q 来摊薄通信开销。
- 偶发卡顿：平均速度可能不差，但等待风险更高。
- 可用性受限：不适合放入严格 deadline 的组。

因此端侧模型新增 `train_time`、`download_time`、`upload_time`、`spike_delay`、`availability_wait`，让服务端从“知道客户端慢”进一步扩展到“知道客户端为什么慢”。

### 6.3 当前维护的聚合状态

`RuntimeStateTracker` 对每个客户端使用滑动窗口，默认最多保留最近 20 条 `ClientRoundReport`。每来一轮反馈就重新计算一次状态，不是每隔 20 轮才计算一次。

当前维护的基础状态：

- `step_time_mean`
- `step_time_std`
- `step_time_cv`
- `round_time_mean`
- `round_time_std`
- `late_rate`
- `early_margin_mean`
- `q_min_hit_rate`
- `q_max_hit_rate`
- `availability_rate`
- `last_staleness`
- `num_reports`

新增的慢因状态：

- `compute_step_time_mean`：最近窗口内 `train_time / local_steps` 的平均值，反映纯计算速度。
- `compute_step_time_std`：纯计算单步耗时标准差。
- `compute_step_time_cv`：纯计算单步耗时变异系数，反映计算侧波动。
- `communication_time_mean`：最近窗口内 `download_time + upload_time` 的平均值，反映通信总开销。
- `communication_time_std`：通信耗时标准差。
- `communication_time_cv`：通信耗时变异系数，反映网络侧波动。
- `communication_ratio_mean`：通信耗时占 `round_time` 的平均比例，反映慢因中通信占比。
- `spike_delay_mean`：偶发卡顿平均耗时。
- `availability_wait_mean`：不可用等待平均耗时。
- `availability_wait_ratio_mean`：不可用等待占 `round_time` 的平均比例。

慢因判断可以先按以下经验规则解释：

- `compute_step_time_mean` 高、`compute_step_time_cv` 低：稳定计算慢。
- `compute_step_time_cv` 高：计算侧波动大。
- `communication_time_mean` 或 `communication_ratio_mean` 高：通信开销主导。
- `communication_time_cv` 高：网络侧波动大。
- `spike_delay_mean` 高：偶发卡顿明显。
- `availability_rate` 低或 `availability_wait_mean` 高：可用性受限。

### 6.4 标准代码添加位置

本阶段只在 `su_compass` 端侧模型中添加，不修改主代码 `src/appfl/scheduler/compass_scheduler.py`。

已经添加的位置：

1. `su_compass/runtime/state.py`

   - 在 `ClientRoundReport` 中新增单轮拆分耗时字段：

```text
train_time
download_time
upload_time
spike_delay
availability_wait
```

   - 在 `ClientRoundReport` 中新增派生属性：

```text
compute_step_time = train_time / local_steps
communication_time = download_time + upload_time
communication_ratio = communication_time / round_time
availability_wait_ratio = availability_wait / round_time
```

   - 在 `ClientRuntimeState` 中新增慢因聚合状态：

```text
compute_step_time_mean/std/cv
communication_time_mean/std/cv
communication_ratio_mean
spike_delay_mean
availability_wait_mean
availability_wait_ratio_mean
```

   - 在 `RuntimeStateTracker._build_state()` 中从滑动窗口的每条 report 提取上述字段并计算均值、标准差、CV。

2. `su_compass/runtime/virtual_runtime.py`

   - `VirtualRuntimeModel.simulate_round()` 原本已经计算：

```text
download_time
train_time
upload_time
spike_delay
availability_wait
```

   - 现在这些拆分耗时会写入 `ClientRoundReport`，使模拟器输出的 `state_snapshot` 包含慢因信息。

后续真正接入主代码时，标准位置应是 `src/appfl/scheduler/compass_scheduler.py` 的 `_record_info()` 之后构造 `ClientRoundReport`。第一阶段仍然只记录 `state_snapshot`，不修改 `_assign_group()`、`_join_group()`、`_create_group()`、`_group_update()` 和聚合器逻辑。

## 7. 默认端侧场景

`build_default_profiles()` 当前提供：

- `stable_fast`：稳定快客户端。
- `stable_slow`：稳定慢客户端。
- `compute_volatile`：计算波动客户端。
- `network_poor`：网络差客户端。
- `availability_limited`：可用性受限客户端。

后续可扩展：

- `important_slow`：高重要度 + 稳定慢。
- `important_volatile`：高重要度 + 高波动。
- `low_utility_fast`：低重要度 + 快客户端。

## 8. 验证方式

完整验证说明见：[`端侧模型验证.md`](./端侧模型验证.md)

8 客户端画像验证（推荐）：

```bash
python -m su_compass.runtime.validate_profiles
```

轻量 JSON 快照验证：

```bash
python -m su_compass.runtime.simulator --mode default
```

单客户端可调验证：

```bash
python -m su_compass.runtime.simulator \
  --mode single \
  --client_id custom_slow \
  --compute_capacity 0.3 \
  --compute_jitter_cv 0.05 \
  --upload_bandwidth_mbps 30 \
  --download_bandwidth_mbps 60
```

预期结果：

- 稳定慢客户端：`step_time_mean` 高，`step_time_cv` 低。
- 计算波动客户端：`step_time_cv` 高，`round_time_std` 高。
- 网络差客户端：`round_time_mean` 高，`late_rate` 更高。
- 可用性受限客户端：`availability_rate` 低于 1，`round_time_std` 更高。
