# 端侧模型接入落地方案

本文档记录 SU-Compass 端侧虚拟时间实验的工程落地方案，是在 [`端侧模型接入FedCompass说明.md`](./端侧模型接入FedCompass说明.md) 基础上的**实施版**说明。

与原文档的差异：

| 原文档侧重 | 本文档侧重 |
|-----------|-----------|
| 主要面向 FedCompass 接入 | **FedAvg / FedAsync / FedCompass 三算法统一虚拟时间标准** |
| 分三阶段（Runner → 观测 → reason-aware） | **分层解耦的工程结构与代码落点** |
| 概念与验证方法 | **具体文件、复用边界、实施顺序、实验入口** |

**文档状态（2026-07-08）**：§5–§8 所述 MVP 工程已落地并通过三算法冒烟测试；§16 记录实现细节与验收结果。

---

## 1. 总体目标

在**尽量不改 `src/appfl/` 主代码**的前提下，建立一套统一虚拟客户端时间实验框架，使后续对比实验满足：

1. **真实训练**：8 个逻辑客户端在单张 RTX 4090 上真实本地训练，产出真实 `local_model` / gradient。
2. **虚拟到达**：下载、上传、卡顿、不可用等待由 `VirtualRuntimeModel` 根据五类端侧画像生成。
3. **统一标准**：FedAvg、FedAsync、FedCompass 使用**同一套**客户端画像、同一套虚拟完成事件、同一套 trace 输出格式。
4. **算法语义不变**：FedAvg 仍等齐聚合，FedAsync 仍单客户端到达即更新，FedCompass 仍 arrival group 半异步；变化的是**时间源**从 wall-clock 变为 virtual time。

推荐论文表述：

```text
We keep the original training, aggregation, model, optimizer, and data partition settings unchanged. The only system-level change is that client arrivals are driven by an event-based virtual runtime model instead of wall-clock execution time.
```

## 2. 实验语义

```text
原始实验：
  真实训练 + 真实到达时间 + 原始算法调度

修改后实验：
  真实训练 + 虚拟到达时间 + 原始算法调度语义
```

具体流程：

```text
一张 RTX 4090
  -> 顺序/分批完成 8 个逻辑客户端的真实训练
  -> 每个客户端得到真实 local_model
  -> VirtualRuntimeModel 生成 virtual dispatch_time / finish_time
  -> 事件队列按 finish_time 排序
  -> server 按 virtual_now 调用对应算法控制器（FedAvg / FedAsync / FedCompass）
  -> 复用 appfl 原始 Aggregator 聚合
  -> 记录 runtime state、scheduler trace、training metrics
```

关键约束（与原文档第 12 节一致）：

- 同一实验内**禁止混用** `time.time()` 与 `virtual_finish_time`。
- **真实训练完成顺序 ≠ server 处理顺序**；server 必须按事件队列中的 `finish_time` 处理。
- 第一版**只记录端侧状态，不改变 Q 分配策略**（不做 reason-aware 调度）。

## 3. 分层解耦架构

为避免代码散乱、便于三算法对比，采用四层结构：

```text
┌─────────────────────────────────────────────────────────────┐
│  实验入口层   su_compass/experiments/                         │
│  run_virtual_fl.py  —  统一 CLI，切换 fedavg/fedasync/compass │
└────────────────────────────┬────────────────────────────────┘
                             │
┌────────────────────────────▼────────────────────────────────┐
│  算法控制层   su_compass/virtual/algorithms/                  │
│  fedavg.py | fedasync.py | fedcompass.py                      │
│  仅负责：虚拟时间下的调度语义（等齐 / 即到即更 / arrival group）│
└────────────────────────────┬────────────────────────────────┘
                             │
┌────────────────────────────▼────────────────────────────────┐
│  虚拟时间核心层   su_compass/virtual/                         │
│  event.py | client_runtime.py | trace.py                      │
│  事件队列、真实训练+虚拟时间桥接、统一 trace 输出              │
└────────────────────────────┬────────────────────────────────┘
                             │
┌────────────────────────────▼────────────────────────────────┐
│  端侧模型层   su_compass/runtime/  （已实现）                  │
│  profile | virtual_runtime | state | validate_profiles        │
└────────────────────────────┬────────────────────────────────┘
                             │ 调用
┌────────────────────────────▼────────────────────────────────┐
│  APPFL 主代码   src/appfl/  （尽量不改）                       │
│  agent | trainer | aggregator                                │
└─────────────────────────────────────────────────────────────┘
```

职责划分：

| 层级 | 目录 | 职责 | 是否依赖 appfl |
|------|------|------|----------------|
| 端侧模型层 | `su_compass/runtime/` | 画像、虚拟时间生成、状态追踪 | 否 |
| 虚拟时间核心层 | `su_compass/virtual/` | 事件循环、训练与虚拟时间桥接 | 是（ClientAgent） |
| 算法控制层 | `su_compass/virtual/algorithms/` | 三算法虚拟调度语义 | 是（Aggregator） |
| 实验入口层 | `su_compass/experiments/` | CLI、配置加载、实验编排 | 是 |

**实验入口统一在 `su_compass/` 下**；`su_compass` 调用 `appfl` 的 trainer 与 aggregator，不在 `examples/serial/` 或 gRPC 真实时间流程中做三算法对比。

### 4.2 不直接复用（在 su_compass 中虚拟化）

| 组件 | 原因 | 替代方案 |
|------|------|----------|
| `SyncScheduler` | 绑定真实「等齐」执行顺序 | `VirtualFedAvgController` |
| `AsyncScheduler` | 绑定真实到达顺序 | `VirtualFedAsyncController` |
| `CompassScheduler` | 依赖 `time.time()` + `threading.Timer` | `VirtualFedCompassController`（逻辑对齐、时间源替换） |
| `examples/serial/run_serial.py` | 同步 FL，不适配半异步与虚拟时间 | `run_virtual_fl.py` |

FedCompass 的 `VirtualFedCompassController` 建议从 `compass_scheduler.py` **复制**以下方法并改造，而非直接改主代码：

```text
_record_info()
_single_update()
_group_update()
_group_aggregation()
_assign_group()
_join_group()
_create_group()
```

改造规则：

| 原始 FedCompass | 虚拟版本 |
|----------------|----------|
| `time.time() - self.start_time` | `self.virtual_now` |
| `threading.Timer` | 向事件队列推送 `FEDCOMPASS_GROUP_DEADLINE` |
| `client_update_time / local_steps` | `report.round_time / local_steps` |

聚合仍调用 `FedCompassAggregator.aggregate()`，与主代码一致。

## 6. 核心模块说明

### 6.1 统一事件 `event.py`（✅ 已实现）

```python
@dataclass(order=True)
class VirtualEvent:
    time: float
    _priority: int          # 同时间下 CLIENT_UPLOAD 优先于 GROUP_DEADLINE
    _seq: int               # FIFO 稳定排序
    event_type: EventType
    client_id: str | None
    payload: dict
```

`EventQueue` 基于最小堆，按 `virtual_time` 弹出事件；**同一虚拟时间下 `CLIENT_UPLOAD` 优先于 `FEDCOMPASS_GROUP_DEADLINE`**，对齐原始 `curr_time > latest_arrival_time` 的边界语义。

事件类型：

| 类型 | 含义 | 使用算法 |
|------|------|----------|
| `CLIENT_UPLOAD` | 客户端虚拟完成并上传 | 全部 |
| `FEDCOMPASS_GROUP_DEADLINE` | arrival group 最晚到达 | FedCompass |

### 6.2 训练与虚拟时间桥接 `client_runtime.py`

`VirtualClientRuntime.train_and_schedule_upload()` 负责：

1. 设置 `trainer.num_local_steps = Q`
2. `client_agent.train()` — 真实 GPU 训练
3. `client_agent.get_parameters()` — 真实 local_model
4. `VirtualRuntimeModel.simulate_round(...)` — 虚拟 finish_time 与 `ClientRoundReport`
5. `RuntimeStateTracker.update(report)`
6. 返回 `VirtualEvent(time=finish_time, type=CLIENT_UPLOAD, ...)`

**虚拟 `train_time` 与真实 GPU 训练耗时无关**；真实训练只产出模型更新，虚拟时间只驱动调度。

### 6.3 算法控制器接口 `algorithms/base.py`

```python
class VirtualAlgorithmController:
    def initialize(self, clients, initial_model): ...
    def on_client_upload(self, event, virtual_now) -> list[VirtualEvent]: ...
    def on_timer_event(self, event, virtual_now) -> list[VirtualEvent]: ...
    def next_dispatches(self) -> list[dict]: ...
    def training_finished(self) -> bool: ...
```

### 6.4 三算法虚拟语义

**FedAvg**（`fedavg.py`）：

- 每轮 8 客户端从同一 `global_model` 开始训练。
- 收集本轮全部 `CLIENT_UPLOAD` 后，在 `virtual_now = max(finish_time)` 处调用 `FedAvgAggregator.aggregate(local_models)`。
- 下一轮从该虚拟时间再派发所有客户端。
- 预期：`network_poor`、`availability_limited` 明显拉长每轮虚拟总时间。

**FedAsync**（`fedasync.py`）：

- 按 `finish_time` 排序处理 `CLIENT_UPLOAD`。
- 每到一个客户端立即 `FedAsyncAggregator.aggregate(client_id, local_model)`，该客户端马上进入下一轮。
- 预期：快客户端更新次数更多，慢客户端 staleness 更高。

**FedCompass**（`fedcompass.py`）：

- 复刻 arrival group、Q 分配、late 处理逻辑。
- `speed = report.round_time / local_steps`，带 `speed_momentum`。
- group deadline 用事件队列替代 `threading.Timer`。
- 预期：快客户端平均 Q 更高，group trace 与论文语义一致。

### 6.5 统一 Runner 主循环 `run_virtual_fl.py`

```text
初始化 clients、profiles、controller、event_queue
下发 initial global_model，首轮 dispatch

while not controller.training_finished():
    # A. 真实训练（串行，不决定 server 顺序）
    while dispatch_queue 非空:
        train + simulate_round -> push CLIENT_UPLOAD

    # B. 虚拟时间推进
    event = event_queue.pop_min()
    virtual_now = event.time

    if event.type == CLIENT_UPLOAD:
        controller.on_client_upload(event, virtual_now)
    else:
        controller.on_timer_event(event, virtual_now)

    dispatch_queue.extend(controller.next_dispatches())

写出 trace，保存 experiment_config.json
```

### 6.6 客户端 ID 约定（✅ 已落实）

- 实验入口统一使用 `"client_0"`…`"client_7"`，与 `build_8client_profiles()` 的 key 一致。
- 与 `RuntimeStateTracker`、`scheduler_trace`、画像布局 `DEFAULT_CLIENT_LAYOUT` 对齐。
- 默认映射见 `validate_profiles.DEFAULT_CLIENT_LAYOUT`。

## 13. 预期实验结论（用于自检）

接入完成后，三算法在**相同虚拟画像**下应能观察到：

1. **FedAvg**：被 `network_poor`、`availability_limited` 拖慢，虚拟 wall-clock equivalent 最长。
2. **FedAsync**：虚拟总时间较短，但 staleness 更明显，快客户端主导更新。
3. **FedCompass**：在 straggler 与吞吐之间折中，group + Q 分配减轻最慢客户端对系统的拖累。
4. **画像一致性**：`summary_table` 与独立验证 `runtime_profile_8clients` 的相对快慢关系一致。
5. **可解释性**：`communication_ratio`、`availability_wait` 等字段可回答「慢在哪里」，为后续 reason-aware 调度提供依据。

## 16. 已完成工程实施记录

本节记录 MVP 的**实现方式**与**验收结果**，便于后续扩展调度策略时对照 baseline。

### 16.1 实现方式概览

按 §9 推荐顺序分 5 步落地：

```text
步骤 1  虚拟时间核心层
        su_compass/virtual/event.py
        → VirtualEvent、EventType、EventQueue（最小堆 + 同时间优先级）

步骤 2  训练与虚拟时间桥接
        su_compass/virtual/client_runtime.py
        → VirtualClientRuntime.train_and_schedule_upload()
        → GPU 真实 train() + VirtualRuntimeModel.simulate_round()

步骤 3  统一 trace
        su_compass/virtual/trace.py
        → TraceWriter：scheduler_trace / group_trace / training_metrics / per-client CSV

步骤 4  三算法虚拟控制器
        su_compass/virtual/algorithms/{base,fedavg,fedasync,fedcompass}.py
        → 复用 Fed*Aggregator；FedCompass 从 compass_scheduler.py 复刻调度逻辑

步骤 5  实验入口与配置
        su_compass/experiments/run_virtual_fl.py
        su_compass/config/virtual_*_mnist8.yaml ×3
```

**主循环**（`run_virtual_fl.py`）：

```text
while not controller.training_finished():
    A. dispatches = controller.next_dispatches()
       for each dispatch:
           agent.load_parameters(global_model)
           train_and_schedule_upload() → push CLIENT_UPLOAD

    B. event = event_queue.pop()          # virtual_time 最小
       virtual_now = event.time
       controller.on_client_upload() 或 on_timer_event()
       新事件（如 GROUP_DEADLINE）入队

    trace.flush()
```

**路径与设备**：创建 `APPFLServerAgent` / `APPFLClientAgent` 前 `chdir(examples/)`，保证 `./model/cnn.py`、数据集路径正确；客户端 `device: cuda` 仅用于真实训练，不参与虚拟调度。

### 16.2 FedCompass 与原始实现的对齐说明

| 维度 | 原始 `CompassScheduler` | 当前 `VirtualFedCompassController` |
|------|-------------------------|-------------------------------------|
| 时间源 | `time.time() - start_time` | `event.time` / `_virtual_now` |
| group deadline | `threading.Timer` | `FEDCOMPASS_GROUP_DEADLINE` 事件 |
| 速度估计 | `client_update_time / local_steps` | `report.round_time / local_steps` |
| Q 分配 | `_join_group` / `_create_group` | 公式一致 |
| 迟到 | `curr_time > latest_arrival_time` → `general_buffer` | 一致 |
| 组聚合 | `general_buffer + group_buffer` → `FedCompassAggregator` | 直接调用同一聚合器 |
| 停止条件 | `_num_global_epochs += len(local_models)` | 一致（客户端更新贡献数） |
| 全局版本 | `global_timestamp` | `get_global_timestamp()` |

**未直接调用** `CompassScheduler` 类（因其绑定 wall-clock 与线程定时器）；调度**语义**对齐，聚合**代码**复用。

### 16.3 虚拟端侧全局参数（冒烟默认值）

冒烟测试**未**使用 GPU 真实耗时或模型字节数驱动调度，而是使用 CLI 默认 + 客户端画像：

| 参数 | 冒烟默认值 | 含义 |
|------|------------|------|
| `base_step_time` | `0.05` s/step | 计算能力 1.0 时每步虚拟训练基准 |
| `model_size_mb` | `5.0` MB | 虚拟下载大小 |
| `update_size_mb` | `5.0` MB | 虚拟上传大小 |
| `seed` | `2026` | 虚拟时间随机种子 |

GPU 仅产出 `local_model`；`finish_time`、到达顺序、迟到判断均由 `VirtualRuntimeModel` + profile 生成。

> 后续严谨实验建议：从 `state_dict` 自动计算模型 MB，并可选校准 `base_step_time`。

### 16.4 冒烟测试结果（2026-07-08）

在 RTX 4090（`device: cuda`）、MNIST 8 客户端、预设虚拟参数下执行：

```bash
python -m su_compass.experiments.run_virtual_fl \
  --algorithm {fedavg|fedasync|fedcompass} \
  --server_config su_compass/config/virtual_<algo>_mnist8.yaml \
  --client_config examples/config/client_mnist.yaml \
  --num_clients 8 \
  --num_global_epochs <budget> \
  --output_dir su_compass/output/smoke_<algo>_mnist8
```

| 算法 | 预算 `--num_global_epochs` | 实际全局模型版本 `global_timestamp` | 客户端更新贡献（FedCompass） | 结论 |
|------|---------------------------|--------------------------------------|------------------------------|------|
| FedAvg | 3 | 3 | — | 同步等齐；`network_poor` 虚拟完成最慢（~20s），拉长每轮 |
| FedAsync | 20 | 20 | — | 快客户端（`stable_fast`）训练轮次明显多于慢客户端 |
| FedCompass | 20 | 13 | 20（达预算停止） | arrival group 生效；快客户端更频繁参与组聚合 |

**FedCompass 计数说明**（与原始主代码一致）：

- `--num_global_epochs`：按**客户端更新贡献数**计预算（单更新 +1，组聚合 +len(local_models)）。
- `global_timestamp`：**server 实际全局模型更新次数**（每次聚合 +1）。
- 故预算 20 可能对应 13 次全局聚合，属预期行为，非调度失败。

**FedCompass group_trace 示例**（冒烟中观测到半异步分组）：

```text
group 0: client_0, client_1, client_4  @ virtual_time ~15.8s
group 2: client_1, client_0              @ ~23.5s
group 1: client_4, client_7              @ deadline ~26.3s
```

**输出目录结构**（三算法同构，冒烟已验证）：

### 16.5 实施中修复的关键问题

| 问题 | 处理 |
|------|------|
| `APPFLServerAgent` 找不到 `./model/cnn.py` | 创建 agent 前 `chdir(examples/)`，配置路径改绝对路径 |
| profile key `client_0` 与 agent id `0` 不一致 | 统一为 `client_0`…`client_7` |
| FedCompass 首轮 8 次单更新后无后续 dispatch | `on_client_upload` 单更新 / 迟到 / 组聚合后均 `append(Dispatch)` |
